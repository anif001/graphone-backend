Important Code Explained

ApiError class (src/utils/ApiError.js) ▶

Extends the native Error class with statusCode, details, and isOperational. The isOperational flag is critical — it tells the error handler whether to expose the error message to the client (operational) or return a generic 500 (programmer error). The Error.captureStackTrace ensures the stack trace starts from where the error was thrown, not from the ApiError constructor.

throw new ApiError(404, 'Company not found');

In the error handler middleware, we check err.isOperational to decide the response shape.

asyncHandler (src/utils/asyncHandler.js) ▶

Wraps an async function and catches any rejected promise, forwarding it to Express's error middleware via next(). Without this, an error thrown in an async route handler would cause an unhandled promise rejection and crash the process.

const getCompany = asyncHandler(async (req, res) => {
  const company = await companyService.findById(req.params.id);
  res.json(new ApiResponse(200, company));
});
Error handler middleware (src/middleware/index.js) ▶

Express error-handling middleware has 4 parameters: (err, req, res, next). The first argument is checked for isOperational. In development, the full stack trace is included in the response. In production, only the message is shown for operational errors, and a generic "Internal Server Error" for programming errors.

if (!err.isOperational) {
  statusCode = 500;
  message = 'Internal Server Error';
  details = null;
}

Common Bugs

  • Missing dotenv call: If dotenv.config() is not called early, process.env will be undefined and the app defaults will be used silently.
  • Middleware order: If express.json() comes after routes, request bodies will be undefined.
  • Not calling next(): If an error handler doesn't call next(err), Express will hang.
  • Express 5 route changes: Express 5 removed string-based regex patterns in routes (e.g., /user/:id(\\d+)). Use path-to-regexp compatible syntax instead.

Debugging Tips

  • Run npm run dev — nodemon auto-restarts on file changes.
  • Check the terminal for Morgan logs: GET /api/v1/health 200 2.882 ms.
  • Use console.log inside middleware to trace the request path.
  • Set NODE_ENV=development to see stack traces in error responses.

Interview Questions

1. Why do we freeze the config object with Object.freeze?
To prevent any part of the application from accidentally mutating configuration at runtime. Once frozen, properties cannot be added, deleted, or changed, ensuring configuration remains predictable and bug-free.
2. What is the purpose of isOperational in ApiError?
It distinguishes expected errors (validation failures, 404s, authentication failures) from unexpected programmer errors (null reference, undefined variable). Operational errors return meaningful messages to the client; programmer errors return a generic 500 with no details, preventing information leakage.
3. What happens if asyncHandler is not used in an async route handler?
If an async route handler throws an error, Express will not catch it because async functions return a promise. The error becomes an unhandled promise rejection, which Node.js will eventually log and crash the process (or, in recent versions, emit an unhandledRejection event).
4. Why is helmet() important in production?
Helmet sets various HTTP security headers (Content-Security-Policy, X-Content-Type-Options, X-Frame-Options, etc.) that protect against common web vulnerabilities like cross-site scripting (XSS), clickjacking, and MIME-type sniffing.
5. How does compression help our API?
It gzip-compresses response bodies, reducing bandwidth by 60-80% for JSON responses. This speeds up API response times for the frontend, especially for large payloads like company details with nested relations.

Key Takeaways

  • Always centralize configuration — never scatter process.env across files.
  • Standardize API responses — frontend teams love predictable shapes.
  • Use asyncHandler to avoid try-catch in every controller.
  • Distinguish operational vs programmer errors for secure error handling.
  • Middleware order matters: security → parsing → logging → routes → 404 → error.

Production Insights

  • In production, set NODE_ENV=production to disable stack traces.
  • Use a process manager like PM2 to keep the server alive on crashes.
  • Consider express-rate-limit (Phase 15) to prevent abuse.
  • Always validate environment variables on startup — crash early if Supabase keys are missing.
  • Use helmet() with custom CSP for frontend domains.

Migration Execution Flow

  1. npm run migrate triggers migrate.js
  2. Reads SUPABASE_DIRECT_URL from config (PostgreSQL connection string)
  3. Creates a pg.Pool connection to the Supabase database
  4. Reads all .sql files from the migrations directory (sorted alphabetically)
  5. Executes each SQL file in order via pool.query(sql)
  6. Logs success/failure for each migration
  7. Closes the connection pool

Why This Schema Design

  • UUID primary keys: Prevent enumeration attacks (vs auto-increment IDs), enable client-side ID generation, and merge-friendly for distributed systems.
  • Slug columns with UNIQUE constraint: Enable URL-friendly lookups like GET /companies/openai instead of exposing UUIDs.
  • Junction tables for M:M: Companies, investors, founders, and news are all many-to-many relationships. Junction tables are the standard normalized approach.
  • ON DELETE CASCADE on FKs: When a company is deleted, its products and funding rounds are automatically removed — no orphaned rows.
  • ON DELETE SET NULL on lead_investor_id: Deleting an investor shouldn't delete funding rounds; the reference just becomes null.
  • CHECK constraint on investor type: Ensures data integrity at the database level — no invalid investor types can be inserted.
  • Updated_at trigger: Automates timestamp management — no application code needed to set updated_at.

Important Code Explained

001_initial_schema.sql — The set_updated_at trigger ▶

This PL/pgSQL function is attached to BEFORE UPDATE triggers on 5 tables. When any row is updated, PostgreSQL automatically sets the updated_at column to the current timestamp before the update executes. This ensures audit timestamps are never stale, even if someone updates the row directly in the database.

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
migrate.js — The migration runner ▶

Uses pg.Pool to connect directly to the Supabase PostgreSQL database via SUPABASE_DIRECT_URL. Reads all .sql files sorted alphabetically and executes them. This is the standard pattern for version-controlled database migrations in production. Unlike ORM-based migrations (like Sequelize), raw SQL gives full control over DDL features like triggers, CHECK constraints, and partial indexes.

Why node-postgres instead of Supabase JS client? The Supabase JS client wraps the REST API and doesn't support executing raw SQL/DDL statements. For schema changes, we need a direct PostgreSQL connection.

Common Bugs

  • Missing SUPABASE_DIRECT_URL: The migrate script will fail if this env var is not set. Get it from Supabase Dashboard → Settings → Database → Connection string.
  • Wrong migration order: If you add a new migration file that depends on a previous one, name it with a higher number (e.g., 002_add_indexes.sql). Files are sorted alphabetically.
  • Foreign key conflicts: When inserting test data, make sure referenced rows exist before inserting rows with foreign keys.
  • UUID vs serial: If you're used to auto-increment IDs, remember that UUIDs are generated client-side or via gen_random_uuid().
  • Slug uniqueness: Two companies with the same name (e.g., "AI" in different contexts) would conflict on slug. Consider appending a suffix in the seed script.

Debugging Tips

  • Run npm run migrate and check the console output — each migration is logged.
  • Use psql or Supabase SQL Editor to inspect tables: \dt lists all tables.
  • Check \d companies to see the full table definition including indexes and constraints.
  • If a migration fails mid-way, run the rollback first, fix the issue, then re-run.

Interview Questions

1. Why UUIDs instead of auto-increment IDs?
UUIDs prevent enumeration attacks (users can't guess IDs), allow offline/client-side ID generation, and prevent ID collisions in distributed/microservice architectures. The tradeoff is slightly larger storage (16 bytes vs 4 bytes for integer) and slightly slower B-tree index performance.
2. What is a junction table and when do you use one?
A junction table (also called an associative or linking table) resolves a many-to-many relationship between two entities. It contains foreign keys to both related tables as a composite primary key. For example, a company can have multiple investors, and an investor can fund multiple companies — this M:M relationship is modeled with the company_investors junction table.
3. What is the difference between ON DELETE CASCADE and ON DELETE SET NULL?
CASCADE automatically deletes child rows when a parent row is deleted (e.g., deleting a company deletes its products). SET NULL sets the foreign key column to NULL when the parent is deleted (e.g., deleting an investor leaves the funding round intact but removes the investor reference). CASCADE is for strong ownership; SET NULL for optional references.
4. Why have a separate slug column instead of using the name in the URL?
Names can contain special characters, spaces, and can change over time. Slugs are URL-safe, permanent identifiers (e.g., "openai" vs "OpenAI"). Using a slug also avoids URL encoding issues and makes for cleaner, SEO-friendly URLs.
5. What is the purpose of the pgcrypto extension?
pgcrypto provides cryptographic functions in PostgreSQL, including gen_random_uuid() which generates random UUIDs. Without the extension, we'd need to generate UUIDs in application code or use a different method.

Key Takeaways

  • Design the schema before writing any API code — the database is the foundation.
  • Use UUIDs for public-facing IDs to prevent enumeration.
  • Junction tables are the standard way to model many-to-many relationships.
  • Always version-control database migrations alongside application code.
  • Database-level constraints (CHECK, UNIQUE, NOT NULL) enforce data integrity even if bugs exist in application code.
  • Index foreign key columns and frequently-queried columns for performance.

Production Insights

  • Never run migrations manually in production — automate them in your CI/CD pipeline.
  • Always test migrations on a staging database first.
  • Consider using pg-migrate or node-pg-migrate for more advanced migration features (transactions, up/down functions).
  • Supabase provides a built-in migration system via their CLI (supabase db push) — consider using it for team projects.
  • For high-traffic tables, consider adding composite indexes for common query patterns (e.g., (company_id, announcement_date) for funding rounds).

Seed Execution Flow

  3. Clears existing data in reverse FK order (junction tables → funding_rounds → products → entities)

Why This Design

  • Slug-based relationships in seed data: Data files reference entities by slug instead of UUID — this makes the data readable and maintainable. The seed script resolves slugs to UUIDs at runtime.
  • Separation of data from logic: Each entity type has its own file. Adding a new company or article doesn't require touching the runner script.
  • Idempotent upserts: Using onConflict with upsert means re-running the seed won't create duplicate rows.
  • Clear in FK order: Deleting in reverse FK order prevents constraint violations. Inserting in forward FK order ensures parent rows exist before children.
  • Service role key: Bypasses Row-Level Security (RLS) during seeding. In production, RLS would be enabled and the service key used only for admin operations.

Important Code Explained

Seed runner FK-safe insertion order ▶

The seed script must respect foreign key constraints. The insertion order is:

1. companies          (no FK dependencies)
2. investors           (no FK dependencies)
3. founders            (no FK dependencies)
4. products            (FK → companies)
5. funding_rounds      (FK → companies, FK → investors)
6. news_articles       (no FK dependencies)
7. company_investors   (FK → companies, FK → investors)
8. company_founders    (FK → companies, FK → founders)
9. company_news        (FK → companies, FK → news_articles)

After inserting entities, the script fetches a slug→UUID mapping from the database. This map is used when inserting related records, avoiding the need to hardcode UUIDs in seed data files.

Upsert with onConflict ▶

Supabase upsert uses the onConflict parameter to specify the unique column for conflict resolution. For most entities, we use slug. For junction tables, we use the composite primary key company_id, investor_id. This ensures the seed is idempotent:

await supabase.from('companies').upsert(company, {
  onConflict: 'slug',
  ignoreDuplicates: false  // false = update on conflict
});

Common Bugs

  • RLS policies blocking inserts: If RLS is enabled, the Supabase anon key can't insert data. Use the service_role_key for seeding.
  • FK violation on funding rounds: If a funding round references an investor slug that doesn't exist in the investors table, the lead_investor_id will be null. The script handles this gracefully.
  • Missing slug map entries: If a product references a company slug that doesn't exist, it's skipped with a warning.
  • Duplicate slugs: Two entities with the same slug will cause an upsert conflict. Ensure all slugs are unique.
  • Data type mismatches: Ensure numeric fields (amount_raised, valuation) are numbers, dates are ISO strings, and all required fields are present.

Debugging Tips

  • Run npm run seed and check the console output for warnings about skipped records.
  • Use Supabase SQL Editor to verify data: SELECT COUNT(*) FROM companies;
  • Check for missing slugs in relationships — the script logs "⚠ Skipping" messages.
  • If a migration hasn't been run, tables won't exist and the seed will fail with "is not present in table".

Interview Questions

1. Why use slug-based references in seed data instead of UUIDs?
Slugs are human-readable and stable. Hardcoding UUIDs makes seed data unreadable and brittle — if you reseed the database, UUIDs change. Slugs let you write { companySlug: 'openai' } instead of { companyId: 'abc-123-...' }.
2. What is the difference between anon key and service_role key in Supabase?
The anon key is for public-facing client requests with RLS enforced. The service_role key bypasses RLS entirely and has full database access. Use service_role for migrations, seeding, and admin operations; never expose it to clients.
3. Why is insertion order important when seeding relational data?
Foreign key constraints require that parent rows exist before child rows. Inserting a product before its company violates the FK constraint. Similarly, when deleting, children must be removed before parents (or use ON DELETE CASCADE).
4. How would you handle seeding 10,000+ records efficiently?
For large datasets, use batch inserts (chunking 500-1000 records at a time) or use PostgreSQL's COPY command for bulk loading. The Supabase JS client can batch via .insert() with an array of records. For very large datasets, use a direct PostgreSQL connection with COPY.
5. What if you need to update seed data after deployment?
Create incremental seed scripts (002_update_data.js) that run after the initial seed. Alternatively, use upsert logic so re-running the seed updates existing records without creating duplicates. Always test seed updates on staging first.

Key Takeaways

  • Seed data should be based on real information, not random fakes — this makes development and testing meaningful.
  • Use slug-based references in seed files for readability and maintainability.
  • Always respect FK constraints — insert parents before children, delete children before parents.
  • Make seeds idempotent — running them multiple times should be safe.
  • Separate seed data (JSON/JS files) from seed logic (runner script).

Production Insights

  • Never use service_role_key in client-side code — it grants full database access.
  • For production deployments, run seeds as part of your CI/CD pipeline, not manually.
  • Consider using Supabase studio or a seed management tool for team collaboration.
  • Always have a rollback plan — test seeding on a staging environment first.
  • Monitor seed execution time; optimize by batching inserts for large datasets.

  companiesService.getFundingRounds(company.id), → funding_rounds ⋈ investors
  companiesService.getInvestors(company.id),     → company_investors ⋈ investors
  companiesService.getNews(company.id),          → company_news ⋈ news_articles
  companiesService.getRelatedCompanies(company.id), → shared investors
])
  ↓
new ApiResponse(200, { company, founders, products, ... })
  ↓
JSON response

Why This Design

  • Lazy database client: The server can start without Supabase credentials configured. The error only surfaces when an endpoint actually queries the DB — useful for development and testing.
  • Service layer separation: All database logic lives in the service. Controllers only handle HTTP concerns (parse request, call service, format response). This makes testing and swapping databases easier.
  • Parallel Promise.all: The 6 relationship queries run concurrently instead of sequentially. This cuts response time from ~600ms to ~100ms (the slowest single query).
  • Related companies via shared investors: Companies are considered "related" if they share investors. This is a simple but effective recommendation algorithm without requiring a separate table.
  • Slug-based routing: URLs are human-readable (/companies/openai instead of /companies/uuid-here).

Important Code Explained

Lazy Supabase initialization (src/config/database.js) ▶

The original design created the Supabase client at require-time, which crashed the server if SUPABASE_URL wasn't set. The lazy pattern defers client creation until the first actual database query:

function getSupabase() {
  if (supabase) return supabase;
  if (!config.supabase.url) {
    throw new ApiError(500, 'Supabase URL not configured...');
  }
  supabase = createClient(...);
  return supabase;
}

Now the health endpoint (GET /health) works without Supabase. Only endpoints that query the database will throw the configuration error.

Parallel data fetching with Promise.all ▶

In getCompanyBySlug, after finding the company, we fetch all 6 relationship sets concurrently:

const [founders, products, fundingRounds, investors, news, related] =
  await Promise.all([
    companiesService.getFounders(company.id),
    companiesService.getProducts(company.id),
    companiesService.getFundingRounds(company.id),
    companiesService.getInvestors(company.id),
    companiesService.getNews(company.id),
    companiesService.getRelatedCompanies(company.id),
  ]);

Each query goes to Supabase independently. The total response time equals the slowest query (typically getFundingRounds or getNews), not the sum of all queries.

Common Bugs

  • Nested sort failure: .order('news_articles(published_at)') may fail if the Supabase REST API doesn't support nested ordering. The fallback returns unsorted results.
  • Missing service_role_key for seeding: The seed script throws immediately if the service role key isn't set.
  • RLS blocking queries: If Row Level Security is enabled, the anon key can't read data. Use service_role_key or disable RLS during development.
  • Slug case sensitivity: Supabase eq is case-sensitive. /companies/OpenAI won't match openai. The frontend should always use lowercase slugs.

Debugging Tips

  • Check the Morgan logs for route matching: GET /api/v1/companies/openai 200
  • Use Supabase SQL Editor to verify data exists before testing endpoints.
  • If getting 404 for a known company, check the slug in the database: SELECT slug FROM companies WHERE name = 'OpenAI'.
  • Add console.time around Promise.all to measure query performance.

Interview Questions

1. Why use Promise.all instead of await-ing each query sequentially?
Promise.all runs all queries concurrently, reducing total response time from the sum of all query times to the time of the slowest query. For a company detail endpoint with 6 independent queries, this can cut latency by 3-5x.
2. How does the related companies algorithm work?
It finds companies that share at least one investor with the target company. First, it gets all investors of the target company. Then it finds all companies those investors have also invested in (excluding the target). The overlap suggests these companies operate in a similar ecosystem.
3. What is the N+1 query problem and does this code have it?
The N+1 problem occurs when you query a list of N items and then make an additional query for each item. Our list endpoint (GET /companies) does NOT have this issue — it's a single query. The detail endpoint makes 7 fixed queries regardless of data size, which is acceptable. A N+1 scenario would be making one query per founder or per product.
4. Why is lazy initialization better for the database client?
It decouples server startup from database configuration. The server can start, serve health checks, and mount routes even when the database is unavailable. The error surfaces only when a database endpoint is actually called, making debugging and deployment more flexible.
5. What tradeoffs exist with slug-based URLs?
Slug-based URLs are more readable and SEO-friendly but less stable than UUID-based URLs — a company rename would require either changing the slug (breaking existing links) or keeping the old slug (inaccuracy). A common pattern is to use immutable UUIDs internally and redirect-friendly slugs publicly.

Key Takeaways

  • Use Promise.all to parallelize independent database queries.
  • Lazy initialization prevents configuration errors from crashing the server.
  • Separate concerns: routes → controllers → services → database.
  • Slug-based lookups provide clean URLs but require careful handling of uniqueness and case.
  • Related content algorithms (shared investors) can be implemented without dedicated tables.

Production Insights

  • Add request timeouts to prevent slow database queries from hanging the server.
  • Cache company detail responses — they're a perfect candidate for Redis caching (Phase 16).
  • Consider pagination cursors instead of offset-based for large datasets.
  • Rate-limit the detail endpoint — it makes 7 database queries per request.
  • Add database connection pooling for production workloads.

Interview Questions

1. How does the type filter work with Supabase?
The query parameter ?type=VC maps to supabase.from('investors').eq('type', 'VC'). This leverages the CHECK constraint CHECK (type IN ('VC', 'Angel', 'Corporate', 'Accelerator', 'Other')) at the database level, ensuring only valid types are queried.
2. Why does getPortfolioCompanies filter out nulls with .filter(Boolean)?
If the Supabase join returns a row where the nested companies data is null (e.g., a company was deleted), we filter it out to avoid returning null entries in the portfolio array.
